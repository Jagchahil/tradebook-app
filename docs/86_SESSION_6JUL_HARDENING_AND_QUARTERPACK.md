## 1. The hardening pair, an honest finding then the real work

Doc 85 listed "Stripe webhook idempotency" and "bank token encryption at rest" as buildable next. On inspection both were already built and wired in the code (doc 81 documents them as done), so doc 85's line had drifted. The genuine gap was that the two most security sensitive pieces in the codebase had zero regression coverage. So the work was to lock them, which is also what feeds the test runner.

**Stripe webhook idempotency.** The claim logic lives in `app/api/stripe/webhook/route.ts`: before any state change it inserts the event id into `stripe_events` with the service role key, and the primary key makes the insert atomic (201 first sight, 409 duplicate, anything else fails open so a signature verified event is never dropped). To test the decision without standing up Next or Stripe, the two pure, side effect free pieces were extracted to `lib/stripewebhook.ts` with no behaviour change: `claimResultFromStatus(status)` (the 201/409/fail open mapping) and `subscriptionRowFrom(sub)` (the row we book money against, preferring the authoritative price over checkout metadata, refusing to build a keyless row). The route now imports both. `test/stripe.test.mjs` covers them, 25 assertions.

**Token encryption at rest.** `lib/crypto.ts` is AES-256-GCM with a safe rollout (a no op until `BANK_TOKEN_KEY` is set, reads handle both `enc:v1:` ciphertext and legacy plaintext). It was already wired through `lib/supabase.ts` (both bank and HMRC token write paths encrypt, read paths decrypt) and `lib/banksync.ts` (decrypts before a sync). `test/crypto.test.mjs` locks it, 28 assertions: round trip, the encryption off no op, legacy plaintext passthrough, fresh IV nondeterminism, wrong key returns null (never throws), tampered ciphertext, auth tag and IV all rejected, malformed input, and every key form (raw base64, hex, passphrase via scrypt). No change to `crypto.ts` itself.

## 2. The one command test runner

`test/run-all.mjs`. Runs every suite in a stable order (all `test/*.test.mjs`, the exams runner, the HMRC runner, and `logic.test.js`), each in its own node process, captures output, reads the per suite "N passed, M failed." line, and prints a compact PASS or FAIL table. On any failure it dumps that suite's full output and exits non zero, so it is safe to wire into CI or a pre push hook. New `*.test.mjs` files are picked up automatically. `logic.test.js` needs the `typescript` npm module to transpile `taxguide.ts` in memory, so when that module is not resolvable (the partial cowork copy has no node_modules) the runner SKIPS it with a clear note rather than reporting a false failure; in the deploy repo, where typescript is installed, it runs. Current result: 14 runnable suites, roughly 3,346 assertions, under a second.

## 3. The quarter end pack

`lib/quarterpack.ts`, pure and framework free, the same discipline as `lib/bankfeed.ts`. It turns confirmed entries plus a target tax year quarter into a shareable summary for an accountant.

- **Quarter maths.** `quarterBounds`, `quarterForDate`, `taxYearLabel` use the standard MTD periods that run from the 6th (Q1 6 Apr to 5 Jul, and so on), so the pack aligns to the MTD quarterly update the customer will owe. String comparison on ISO dates, no timezone traps.
- **The builder.** `buildQuarterPack` computes the quarter itself (the MTD update content: trade and property streams split by `income_type`, income, expenses by category sorted, CIS suffered) and the year to date position up to the quarter end. The running tax estimate comes from the canonical engine (`soleTraderTax`, `class2Voluntary`) applied to year to date trade profit only, with property profit shown separately and excluded, because property is taxed on its own schedule (its own rates from April 2027). The MTD test is on gross qualifying income, trade plus property before expenses, via `mtdForIncomeTaxRequired`. One source of numbers, never hand rolled.
- **The document.** `renderQuarterPackHtml` returns a complete, branded, print styled HTML document (Inter, the Lekhio indigo, `@media print` rules, a Save as PDF button hidden in print). There is no server side PDF library in this codebase by design; documents are the browser's own print to PDF, exactly as the invoice pages already work. The copy holds the doctrine: prepared from confirmed entries and the published HMRC figures, for the user and their accountant, not a submission to HMRC. Business name is HTML escaped.
- **Delivery.** `GET /api/quarter-pack` authenticates with the caller's own Supabase token (the `account/export` pattern), reads only that account, confirmed rows only, and returns the document (or the structured pack with `?format=json`). Query `?year=2026&q=1..4`, defaulting to the quarter today falls in. Private, no store, noindex. Two small service role helpers were added to `lib/supabase.ts`: `getConfirmedTransactionsForRange` and `getBusinessName`. `test/quarterpack.test.mjs`, 51 assertions, including that the estimate matches the engine to the penny and that the HTML carries no banned dash characters.

## 6d. CI fix: parity suites skip in single-repo CI

Switching CI to `node test/run-all.mjs` (commit `34c5687`) surfaced a latent issue: the runner runs the four `*-parity` suites, which the old CI never ran. Those suites compare this repo's engine against the mobile app's engine at a sibling path `../tradebook-app/lib/tax.ts`. That sibling exists locally (both repos sit side by side) but not in a single-repo GitHub checkout, so CI failed with `ENOENT lib/tax.ts` on all four (ltd, nisl, property, tax parity) while everything passed locally and on Vercel. Fix: `run-all.mjs` now skips the `*-parity` suites when that sibling engine is absent, the same graceful skip it already uses for `logic` without `typescript`. Proven by reproducing the CI layout (a lone repo, no sibling): parity and logic SKIP, everything else passes, exit 0. Where both repos exist the parity guard still runs in full (2,778 assertions). Note: the parity guard therefore does not run in the web repo CI; if we ever want it there, check out the app repo alongside in the workflow. The lockfile regen (`deed0a3`) was a red herring for this, harmless to keep.

## 6e. AI cost governance and the autonomy dial (BUILT 7 Jul)

Two pure, tested engines that answer Jag's "don't let credit blow up" and "make the agent actually DO things safely" asks.

**Cost governance, `lib/aicost.ts` (21 tests).** Five rings inside the Anthropic console monthly cap: (1) a kill switch, `AI_KILL_SWITCH=on` disables every AI call instantly with no deploy, wired live now through `lib/claude.ts` `ready()` so every call and `hasClaudeConfig()` respect it; (2) a global daily cap; (3) a global monthly cap; (4) a per user daily cap; (5) model tiering plus `estimateCostPence` for visibility, rounded up so an estimate is never rosier than the bill. `decideSpend(counts, caps)` is the pure decision (kill and global stops win before the per-user cap); `resolveCaps` reads `AI_GLOBAL_DAILY` (5000), `AI_GLOBAL_MONTHLY` (60000), `AI_USER_DAILY` (40) with safe defaults. The live per-call counters use the existing `ai_usage` table and `bumpAiUsage`. Wiring `decideSpend` as the single chokepoint in every AI route is the next step, done as each AI feature comes online (all credit gated); the kill switch and the existing per-feature `ASK_*`/`DRAFT_*` caps plus the console cap already make a runaway bill impossible.

**The autonomy dial, `lib/autonomy.ts` (33 tests).** The dial the customer sets (`suggest` | `draft` | `auto`, stored on `users.autonomy_level`, default `suggest`). The doctrine is encoded in code, not a prompt: every action is classified `admin` (reversible), `prepare` (builds toward an irreversible step) or `irreversible` (file, pay, move money, buy, message a third party), an unknown action defaults to `irreversible` so nothing slips past. `decideAction(action, level)` returns `{ mode, requiresApproval, actionClass }`, and the exhaustive test proves the invariant: NO level, for ANY irreversible or unknown action, ever auto-executes or skips approval. `auto` only ever runs reversible admin; money and filing always need a human yes. This is the spine that lets the agent do everything up to the legal line and leave only the tap, without ever being able to cross it.

**The dial is now a real app setting (wired 7 Jul).** A "How much Lekhio does for you" section on the app Settings screen (`app/(tabs)/settings.tsx`) with three tappable options, Suggest, Prepare, Auto, each with plain copy and a check on the active one, plus a footnote "anything that files to HMRC or moves money always asks you first." It reads and writes through app helpers `getAutonomy`/`setAutonomy` (`tradebook-app/lib/supabase.ts`) which call the new authed `GET`/`POST /api/autonomy` route (Bearer token, `verifyAccessToken`), which persists to `users.autonomy_level` via `getAutonomyLevel`/`setAutonomyLevel`, every write validated through `parseLevel` so only a real level lands. The column SQL already ran on prod. Next, route the agent's first real `auto` admin action through `decideAction` when there is one to route (still zero credit); today the dial is a live, persisted user choice ready for the agent to read.

## 6g. The tax optimiser engine (BUILT 7 Jul) — the "hands" for the dial

`lib/taxoptimiser.ts` (34 tests): the "lower my tax as much as possible" engine, pure and deterministic, zero credit. `findOptimisations(input)` scans a user's YTD figures + profile and returns every LEGITIMATE lever, richest saving first: pension to step out of the 40% band, AIA timing on a purchase goal (saving = cost x marginal rate), use-of-home flat rate, mileage-vs-fuel prompt, missed common expense categories, a CIS refund flag (information), an honest incorporation answer, and a property-costs prompt. `marginalRate()` mirrors the agent's bands (0/0.26/0.42/0.62). `totalEstimatedSaving()` is the headline. `applyDial(opts, level)` maps each lever through `decideAction`, and the doctrine test proves the money levers (pension `make_payment`, buying `purchase`) are always requiresApproval and never `auto` at any level, while reversible admin (home office, logging) can auto-run at the `auto` setting. This is the engine that makes the autonomy dial actually DO something. WIRED SAME SESSION: `getOptimiserInput(userId)` in `supabase.ts` assembles the input from the user's confirmed tax-year data + profile (trade stream, categories, CIS, employment income, purchase goal, property income/expenses split by `income_type`; home-office and mileage inferred from categories); the authed `GET /api/optimise` returns `{ level, totalSaving, optimisations }` with the dial already applied; the app has `getOptimisations()` and a "Ways to save" screen (`app/ways-to-save.tsx`) reached from a lead NavRow in the Money tab, showing the headline total, each lever with its saving and a plain badge (Lekhio handles this / Ready when you are / Needs your yes / Good to know), and a line pointing to the dial in Settings. The engine reuses only the canonical taxengine and ltdengine, so its numbers are parity-safe.

**Incorporation, answered honestly not reflexively.** The lever calls `ltdengine.compare(projTradeNet)` and tells the user the truth from our own maths. On 2026/27 rates a sole trader who draws all their profit out is genuinely better off at every profit level (dividend rates up 2pp, employer NI at the £5k threshold, corp tax 19-25%), so at a live-question profit (>= £50k) the lever says "Should you go limited? Not yet", names the gap they would LOSE, and flags the one condition that flips it (leaving money in the business), pointing to the free comparison tool and an accountant. If the maths ever favours a company for that user, the message flips to "a limited company could save you tax". `estSaving` is always 0 (information, never summed into the headline); the pound figure lives in the words. This is the compliance line held in code: we never push a structure change our own numbers say loses money.

**Property costs.** Rental income with almost no expenses logged (< 10% of rents) surfaces a prompt to log mortgage interest (the 20% credit), repairs, agent fees and insurance, a reversible `log_entry` admin lever.

## 6h. The proactive tax-saving signal (BUILT 7 Jul) — Rakha pushes the biggest missed claim

The optimiser was pull-only (the "Ways to save" screen); this makes it proactive without duplicating what the agent already surfaces. The agent already pushes pension (as `higher_rate_approach`/`pa_taper`), AIA (`aia_timing`) and missed common expenses (`expense_completeness`). The one quantified optimiser lever it did NOT push is use-of-home, missed by almost everyone. New agent signal `home_office_saving` (`lib/agent.ts`, +7 agent tests, 187 total): fires as a card once a year, after four months of data, when there is profit, a positive marginal rate, and no "home" category logged (read the same way `expense_completeness` reads categories, so NO new aggregate or RPC is needed). It quantifies the saving exactly as the optimiser does (`homeOfficeFlatRateMonthly(25) x 12 x marginalRate`) so the figure matches the Ways to save screen to the penny, and the copy holds the doctrine ("a suggestion from your numbers, not advice, you decide"). It stays silent for anyone already claiming and for old RPCs. Full suite green (3,518 assertions), engines typecheck clean under strict; no SQL.
