## 1. Fixed in this pass (code, shipped)

- **Tax: WDA main-pool rate 18% → 14%.** HMRC reduced the main-pool writing-down allowance to 14% from 6 April 2026. `lib/taxengine.ts` corrected, and the exam bank updated so the suite tests the right value. Exam suite: **87/87 pass.** (Special-rate pool stays 6%; everything else, income tax, NIC, dividends, CGT, VAT, mileage 55p, CIS, MTD thresholds, corporation tax, verified correct against GOV.UK.)
- **Security (CRITICAL): removed hardcoded HMRC OAuth state secret.** `lib/hmrc.ts` no longer falls back to a literal `'lekhio-hmrc-state'` (which is in the public repo and would let anyone forge a `state` for any user). It now requires `HMRC_STATE_SECRET` (or the server-only service-role key) and fails closed if absent.
- **GDPR delete now reports real success/failure.** `deleteUserData` checks every sub-delete; erasure no longer returns "success" if a table delete silently failed.
- **PostgREST filter hardening.** Added `encodeURIComponent` to the two remaining raw-interpolated queries (`getPhoneForUser`, `weeklyTotals`).
- **`/api/draft-invoice` locked down.** CORS restricted from `*` to our own web origin (native apps ignore CORS anyway), added a durable per-IP daily cap, and lowered the global daily ceiling 2000 → 500.
- **MTD category mapping.** Added `use of home` / `use_of_home` aliases so home-office entries map to `premisesRunningCosts` not `otherExpenses`. (The new `wages`→`wagesAndStaffCosts` and `subcontractor`→`paymentsToSubcontractors` mappings were already correct.)
- **App: transaction editor category list unified** with the Add screen, so a `wages`/`subcontractor`/`travel`/`phone` entry can be re-selected when editing.
- **SEO: added `robots.ts` and `sitemap.ts`** (both were entirely missing). The sitemap auto-includes all 12 trade long-tail pages from the single `TRADES` source.

## 4. Scale to 20,000 users, what breaks first

Verdict: **the data model, indexes, idempotency, cost caps and cron are already built for scale.** One architectural fix is needed before high volume, plus durable rate limiting.

1. **Webhook is synchronous (the #1 scale fix).** `app/api/whatsapp/route.ts` does all AI/Whisper/DB work before returning 200. Meta needs a 200 in ~5s or it retries. With AI **off today** the regex paths (mileage, money parser, CIS) are fast, so this is not biting now, but before turning AI on at volume, restructure to: verify + claim + return 200 immediately, then process in `after()` / `waitUntil` or a queue, and set `maxDuration`. This is the single thing to do before scaling AI.
2. **Rate limiting is in-memory per instance.** The burst limiter (`lib/ratelimit.ts`) won't hold across serverless instances. Move to Upstash Redis (the durable AI cap already protects spend; this protects against reply-storms).
3. **Weekly-summary cron is N+1** (one query per user). Replace per-user `weeklyTotals` with a single grouped Postgres aggregate before the user base is large, or it will time out mid-fan-out on Sundays.
4. Ensure Supabase REST is on the connection pooler at high burst.

## 5. App correctness follow-ups (not blocking, do before relying on quarter accuracy)

- **Figures key off `created_at`, not `transaction_date`.** Dashboard/Tax/CIS/year-summary screens filter by insert date, not the receipt's real date. Harmless today (WhatsApp entries set both to "now"), but once AI receipt-parsing reads past dates from photos, a back-dated receipt could land in the wrong tax quarter. Switch the filters to `transaction_date ?? created_at` and book paid-invoice income at the invoice date.
- **Transaction detail: add an income/expense direction toggle** so a mis-parsed entry can be corrected and confirming can never flip its sign.
- Surface load errors on the dashboard/tax screens (currently a failed read shows as an empty "nothing yet" state).
- Make the phone-normalization parity (app `toUkE164` ↔ web `normalizeUkPhone`) structural, a shared module or a parity test, so a future edit can't split a user's identity again.

## 6. Competitors, where Lekhio wins, where it's exposed

**No HMRC-recognised competitor offers true WhatsApp-first capture.** The only WhatsApp-native rivals (Accounted/Penny, TaskDrop, Snapfile, Snyp) are pre-launch, capture-only, or unrecognised. Incumbents (Coconut, QuickBooks Sole Trader, FreeAgent, Xero Simple, Sage, ANNA, untied) treat WhatsApp only as a channel to send a finished invoice.

**Wins:** WhatsApp photo + voice capture (the moat), voice notes as primary input, trade-first CIS done properly, "claim by text" reliefs, one all-in price, explicit approval gate.

**Exposed (table stakes rivals already have):** HMRC recognition (top risk, gates filing), Open Banking bank feeds, GPS mileage, accountant access portal, app-store reviews/proof, and "free" competition (ANNA files MTD ITSA free; FreeAgent free via NatWest/Mettle).

**Positioning for the April 2026 MTD mandate wave:** lead with the mandate, sell time-on-site ("text it, done") against free-but-fiddly rivals, make CIS + refund tracker the trade hook, and keep the honest "file to HMRC, soon" marker until recognised.

**Prioritized roadmap:**
- **Now:** start HMRC MTD recognition (process, not code, begin the day the Ltd exists); ship read-only Open Banking feed; mandate-led landing page; receipt-OCR correction loop in WhatsApp.
- **Next:** GPS/auto mileage; accountant read-access portal; MTD pack export to a recognised filing path as a bridge; seed app-store reviews.
- **Later (post-recognition):** direct file-to-HMRC with the live approval gate; live tax-owed balance from the bank feed; multi-year history + Final Declaration.
