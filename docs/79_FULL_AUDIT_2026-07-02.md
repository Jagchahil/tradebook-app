## 2. Security findings

### No critical issues, no leaks
The two catastrophic paths are correct. The WhatsApp webhook verifies the `x-hub-signature-256` HMAC over the raw body with a constant time compare before any side effect, and fails closed if the secret is unset. The Stripe webhook verifies the signature with a 300 second replay window and constant time compare before any state change. The HMRC submit paths every one throw unless `approved === true` and record a durable approval row before the network call, and stay sandbox only. Bank and HMRC tokens are encrypted at rest with AES-256-GCM and a fresh random IV per call, stored in service role only tables with RLS and no policies, and never returned to any client. PostgREST filters are consistently wrapped in `encodeURIComponent`. Message content, phone numbers and secrets are kept out of logs. The phone binding trigger stops a client pointing its account at someone else's number. IDOR was checked on the invoice, pay, export, delete and transaction mutation paths and every one scopes to the caller's own id.

### Fixed this pass
- **No global security headers (was HIGH).** The site shipped with no HSTS, no X-Frame-Options, no nosniff, no Referrer-Policy and no CSP, which left the public invoice and pay pages clickjackable and offered no transport hardening. A `next.config.mjs` now sets all of these globally. The CSP is deliberately scoped to `frame-ancestors 'none'; base-uri 'self'; object-src 'none'`, which closes clickjacking and base tag injection without risking a broken render. A fuller script and style CSP should be added and tested in a Vercel preview as a follow up.
- **Open pay endpoint had no rate limit (was HIGH).** `/api/pay/[id]` would spin up a Stripe Checkout for any unpaid invoice id with no throttle, usable to mass create sessions. It now rate limits per IP and, on the limit, sends the customer back to the invoice rather than erroring, so a genuine payer is never blocked.
- **Invoice email link not escaped (was MEDIUM).** The link in the invoice email was interpolated without escaping, unlike every other field. It is server built today so not exploitable, but it is now validated as an https URL and escaped, closing the latent injection sink.
- **Bank callback referrer leak (was LOW).** The TrueLayer callback URL carries the one time auth code and signed state. The callback page now sets `Referrer-Policy: no-referrer` so that URL can never leak through a Referer header.

### Operational, not code (do before real users)
- **Reject anonymous sessions.** The code to reject anonymous Supabase sessions is present and correct but gated behind `REJECT_ANON_USERS`. It is left as a flag rather than flipped in code, because flipping it wrongly could lock users out, and it is a one line, instantly reversible env change. Set `REJECT_ANON_USERS=true` in Vercel and confirm anonymous sign in is off at the Supabase project before launch. This is the single highest priority operational action.
- **Dedicated signing secrets.** The HMRC state and lead token HMAC secrets fall back to the Supabase service role key when their own env vars are unset. This fails closed and is not a hardcoded default, but it couples two trust domains. Set a dedicated `HMRC_STATE_SECRET` (already set per handover) and a `LEAD_TOKEN_SECRET` in production and drop the fallback later.
- **Shared rate limiter.** The burst limiter is in memory per instance, so it does not hold across Vercel instances. The durable per key daily caps are the real backstop and they do hold, so AI and message spend are protected. Move the burst limiter to Upstash before turning AI on at high volume.

## 3. Mobile app findings

The app scan was clean on security: no service role key or any non `EXPO_PUBLIC_` secret anywhere, `toUkE164` is byte identical to the web normaliser, and the app throws on anonymous sign in in production. CIS is handled correctly everywhere as tax already paid, never as an expense. Most tax maths already defers to the real engine.

The issues found were on the money screens, where a few figures still used a flat tax percentage instead of the real 2026/27 engine. All are fixed:
- **Quarterly summary set aside used a flat 30% (was the worst one).** This is the figure the user approves before MTD prep. It now annualises the quarter profit, runs it through the real engine so the personal allowance and correct bands apply, takes a quarter of that and nets off any CIS. The copy no longer claims a fixed 30%.
- **Dashboard tax saved used a flat 26 or 42%.** Now derived as the difference between the engine tax on income and the engine tax on profit, so it reflects the user's true marginal rate.
- **Tax tab quarterly tax saved used the same flat bands.** Now computed from the engine on annualised figures.
- **Activity list grouped entries by the logging date, not the receipt date.** A back dated receipt now files under the day it happened, matching the tax period logic used everywhere else.
- **Accountant wording softened for honesty.** The screen and dashboard card no longer say "Your accountant", "Expert UK tax help" or "a real answer", which implied a qualified human. They now read "Ask Lekhio" and "a straight answer from the UK tax rules", with the existing disclaimer intact.
- **Transaction detail colours realigned to the theme.** The green and red for amounts drifted from the theme tokens, so the same amount showed a slightly different shade than the rest of the app. Fixed.

What is genuinely good in the app: honest approval gate messaging throughout, correct CIS treatment and refund framing, good loading, error and empty states, and only confirmed entries feeding money and tax figures.

## 5. Competitor benchmark, July 2026

**Where Lekhio is genuinely ahead:** not one of the ten incumbents or challengers checked (Coconut, QuickBooks Sole Trader, FreeAgent, Xero Simple, Sage, ANNA, untied, Crunch, GoSimpleTax, TaxScouts) offers WhatsApp or voice note capture. All are app plus bank feed plus receipt scan. Trade native identity with CIS done properly, and the explicit "we prepare, you approve" gate marketed as a principle, are also differentiators none of them lead with.

**Where Lekhio is behind, and it is the honest part:**
- **HMRC recognition.** From April 2026, 864,000 sole traders and landlords are legally mandated. The recognised competitors can serve them and Lekhio cannot yet. The one saving grace is that the closest WhatsApp first rival, TaskDrop, is also stuck in sandbox awaiting HMRC production access, so whoever lands WhatsApp first plus recognised first owns the category. It is a winnable race.
- **Live Open Banking feeds.** Now table stakes. A subcontractor with 200 transactions a quarter cannot photograph their way to a complete quarterly update. WhatsApp capture should be positioned as the thing that enriches and approves a bank feed, not a replacement for it.
- **The price floor is zero.** FreeAgent is free via NatWest and Mettle, Sage has a free tier, and ANNA now files MTD free and is recognised. Lekhio must not compete on price. Compete on "free tools make you do the work, Lekhio does the work and you just approve it".
- **CIS depth.** FreeAgent and Crunch cover contractor and subcontractor CIS. Lekhio needs CIS to be excellent, not just present, and should own the refund narrative, since CIS subcontractors are structurally owed money.

**The five highest leverage moves, ranked:** 1) land HMRC production recognition, nothing else matters until it ships; 2) ship live bank feeds designed as the spine WhatsApp rides on, with a frictionless WhatsApp reconsent flow at the 90 day boundary; 3) make CIS best in class and own the "watch your refund build up" story; 4) turn the approval gate into the lead trust message against agentic rivals like ANNA that market "AI does your taxes"; 5) repoint the free web tools at CIS refunds specifically, which is where the emotional and financial energy is.

**Ideas from newer entrants worth considering:** customer pays the card fee invoicing (Manano, TaskDrop) turns invoicing into a free get paid faster feature; multilingual voice capture (Manano), cheap to add on the existing Whisper pipeline and a real differentiator; and a live "here is your number" money position front and centre rather than buried.

Caveat carried from the research: HMRC removed its static recognised software list in July 2025 and replaced it with a JavaScript finder tool, so most competitor recognition claims rest on vendor statements plus trade press. Only untied was directly confirmed on the live finder.
